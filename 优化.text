已实现策略（在尽量不牺牲召回/准确的前提下省成本、提速）
主检索 + 申诉锚点 +（按需）等级锚点并行
用 ThreadPoolExecutor 同时跑 history_aware_retriever、申诉多路锚点、
以及（在未跳过时的）等级多路锚点，缩短端到端 wall-clock；单轮里向量编码次数与串行一致，
主要省的是等待时间。

单类锚点内部多查询并行
等级/申诉各自的 _parallel_fetch_anchor_queries 对多条固定问句并行 invoke，
并发度受 RETRIEVAL_PARALLEL_WORKERS（默认 8）与问句条数共同限制，
同样偏向 延迟 而非少算检索。

条件性跳过等级锚点（GRADE_ANCHOR_CONDITIONAL，默认开）
仅当用户句子里像申诉（_APPEAL_TRIGGER）且不像等级/分数场景
（无 _GRADE_SCORE_TRIGGER 命中）时，不跑等级多路锚点，减少无效 embedding/相似度计算；
一旦同时提到等级、分数、考核等，仍会跑全量等级锚点，避免误伤「申诉+扣分/等级」类问题。

条件性跳过等级池回退（GRADE_POOL_CONDITIONAL，默认开）
用户问题不含 _GRADE_SCORE_TRIGGER 时，不执行 GRADE_POOL 的那次额外 similarity
（省一次批量检索）；仍保留「合并结果里已有等级+分数映射则本来就不补池」的原逻辑。

本地嵌入可选 GPU（EMBEDDING_DEVICE=auto/cpu/cuda，在 config.py + embeddings_util.py）
多路检索时编码更密集，auto 在可用 CUDA 时用 GPU，
通常能明显降低 本地 嵌入耗时（API 成本不受影响）。

检索链路策略：
单阶段：用什么信号检索
稠密检索（Dense / 向量）：query 与 chunk 的 embedding 相似度，你项目里的 Chroma + MMR 就属于这一类变体。
稀疏检索（Sparse / 关键词）：BM25、倒排索引，擅长专有名词、条款编号。
混合检索（Hybrid）：向量 + BM25，再 加权融合 或 倒数排名融合（RRF），兼顾语义与字面匹配。
2. 多阶段：先多取、再收紧
Retrieve-then-Rerank（粗排 + 重排）：先 fetch_k 大一些，再用 cross-encoder / 商用 rerank API 对 Top-N 重排序。
级联（Cascade）：先用很便宜的检索缩小范围，再用更贵的模型只在子集上算。
3. 查询侧策略（让检索更「对题」）
查询改写 / 扩写：用语义等价或 LLM 生成「更好搜」的问句（历史感知检索也是一种改写）。
Multi-query：一个问题拆成多条子查询，检索后合并去重。
HyDE：先让模型假写一段「理想答案」，用这段的向量去检索（实现成本和风险要评估）。
4. 文档侧与索引结构
父子块 / 小块检索大块返回：检索用小 chunk 提高命中，拼接时带回父段落上下文。
元数据过滤（Metadata filter）：按部门、年份、文档类型先筛再向量检索。
图检索 / 结构化增强（Graph RAG）：实体—关系图扩跳检索，适合复杂依赖关系（成本更高）。
5. 融合与去重
RRF、加权分数、倒排表融合：多路召回结果合成一个排序。
MMR：在相关性与多样性之间折中（你已在使用 MMR）。
6. 检索后处理
上下文压缩：只保留和问题最相关的句子/片段再进 prompt。
多跳 / Agent 检索：模型根据中间结果决定再查什么（更慢、更贵，要控权限与预算）。
